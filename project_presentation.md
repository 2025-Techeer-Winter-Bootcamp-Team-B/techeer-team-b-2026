## 1. 서비스 소개

**SWEETHOME**은 **부동산 자산 관리와 시장 분석을 한 번에 지원하는 데이터 분석·시각화 서비스**입니다.  
국토교통부, 한국부동산원 등 공공/외부 데이터를 수집·정제하고,  
사용자의 보유 자산과 관심 아파트 정보를 함께 분석하여 **“지금 내 자산이 시장에서 어느 정도 위치에 있는지”** 직관적으로 보여줍니다.

- **타깃 사용자**
  - 부동산을 1채 이상 보유하고, 자산 현황과 시장 흐름을 함께 보고 싶은 사용자
  - 내 집 마련을 준비하면서, 여러 아파트를 비교하고 싶은 사용자
- **핵심 목표**
  - 복잡한 표·숫자 대신 **시각화된 통계**로 시장을 이해하게 만들기
  - **내 자산·관심 아파트·시장 통계**를 한 화면에서 비교할 수 있게 만들기

---

## 2. 핵심 기능 소개

### 2-1. 홈 대시보드 – 내 자산 & 시장 요약

홈 화면에서 **내 자산과 관심 아파트, 시장 통계가 한눈에 보이도록** 구성했습니다.

- **내 자산 등록 및 관리**
  - 내가 보유한 아파트, 매입가, 면적, 매입일 등을 등록
  - 백엔드에서 실거래 데이터를 바탕으로 **현재 추정 시세** 계산
  - 자산별 평가 손익, 수익률, 시세 변화 추이 제공
- **관심 아파트 관리**
  - 관심 목록에 아파트를 추가하고, 시세/거래량/랭킹을 함께 확인
  - 실제 사용 중 문제였던 “0억으로 나오는 시세”를 개선하여,
    - 최근 거래가 없더라도 과거 거래/평균 시세를 활용해 **현실적인 추정 시세**를 제공
- **뉴스 / 시장 정보**
  - 부동산 관련 주요 지표와 함께, 시장 이해를 돕는 정보 카드 구성

→ 발표 포인트: **“홈 한 화면에서 내 자산과 시장을 동시에 본다”**에 집중해서 설명하면 좋습니다.

---

### 2-2. 비교 기능 – 여러 아파트를 한 번에 비교

여러 단지를 동시에 비교하면서, **어디가 더 나에게 유리한 선택인지** 볼 수 있는 기능입니다.

- **다수 비교**
  - 1:1 비교뿐만 아니라 **여러 아파트를 동시에 선택**해서 비교
  - 가격, 평당가, 거래량, 랭킹 등 여러 지표를 **막대 그래프/차트**로 시각화
- **지표 중심 설계**
  - 단순 시세 비교를 넘어서,
    - 평당가
    - 거래량 추이
    - 지역/전국 랭킹
    - 변동률
  - 등의 지표를 기준으로 판단할 수 있도록 구성

→ 발표 포인트: “**감(갬성)**이 아니라, **데이터 기준**으로 단지를 비교할 수 있게 해준다”고 강조하면 좋습니다.

---

### 2-3. 통계/시장 분석 – 시장을 한눈에 보는 대시보드

전체 시장의 흐름을 이해할 수 있는 **통계 대시보드**를 제공합니다.

- **전국/지역별 통계**
  - 전국/지역별 평당가, 거래량 추이
  - 특정 시점/기간을 선택해 **시계열 분석**
- **지수 기반 분석**
  - 부동산 지수(HPI) 등 지표를 시각화해 **상승장/하락장 구간을 직관적으로** 확인
- **시각화 중심**
  - D3.js, Highcharts 등을 활용한 **인터랙티브 차트**
  - 툴팁/필터/범례로 사용자가 원하는 구간만 골라서 볼 수 있음

→ 발표 포인트: “**뉴스에서만 보던 지표들을, 내 자산 문맥에서 같이 볼 수 있다**”라고 연결해 설명하면 좋습니다.

---

### 2-4. 지도 기능 – 클러스터링과 공간 기반 탐색

지도 기반으로 아파트를 탐색하고, **공간 정보와 함께 단지를 이해**할 수 있습니다.

- **지도 기반 검색**
  - 카카오 지도 API 기반으로, 특정 지역/단지를 지도에서 바로 검색
  - 지도 범위를 움직이면, 해당 영역의 아파트 목록이 자동 갱신
- **클러스터링**
  - 단지 수가 많은 지역에서도 **클러스터링으로 성능과 가독성 확보**
  - 줌 레벨에 따라 집계 단위가 달라지며, 한눈에 밀집 지역을 파악
- **상세 정보 연계**
  - 지도에서 단지 선택 시, 상세 페이지로 이동하여
    - 실거래 내역, 가격 추이, 면적별 시세, 인근 시세 비교 등을 함께 확인

→ 발표 포인트: “**지도 = 위치만 보는 도구**가 아니라,  
**데이터 분석의 입구 역할을 하는 UI**로 설계했다”고 말해주면 좋습니다.

## 3. 데모 영상(30초) 구성 가이드

30초 안에 **전체 흐름이 보이도록** 아래 순서를 추천합니다.

1. **(5초) 홈 대시보드**
   - 내 자산 카드 + 관심 아파트 카드 + 간단한 시장 통계가 한 화면에 보이는 장면
2. **(10초) 자산 상세**
   - 특정 아파트 상세 페이지로 들어가
   - 실거래 내역, 가격 추이 그래프, 면적 필터, 기간 필터 등 **그래프와 표가 동시에 있는 장면**
3. **(10초) 비교 기능**
   - 여러 아파트를 선택해서 막대 그래프로 비교하는 화면
   - “어느 단지가 더 비싼지/저렴한지” 한눈에 보이는 장면
4. **(5초) 지도 화면**
   - 클러스터링이 적용된 지도에서 줌 인/아웃하면서 단지가 모였다 흩어지는 모습

→ 이 흐름대로 녹화하면, **서비스가 뭘 하는지 직관적으로 이해되는 30초 소개 영상**을 만들 수 있습니다.

## 4. 시스템 아키텍처 요약

이 프로젝트는 **Frontend / Backend / Database / 인프라 & 모니터링**으로 나뉜 풀스택 구조입니다.

- **Frontend**
  - React + TypeScript + Vite
  - Tailwind CSS로 UI 구성, Highcharts/D3.js로 데이터 시각화
  - Clerk를 이용한 인증/인가, Axios로 백엔드 API 연동
- **Backend**
  - FastAPI 기반 REST API 서버
  - SQLAlchemy + Pydantic으로 데이터/스키마 관리
  - 실거래/통계 데이터를 수집·가공하는 배치/스크립트 존재
- **Database & GIS**
  - PostgreSQL + PostGIS로 공간 데이터(위도/경도, 단지 위치) 관리
  - Redis로 캐싱 적용 (자주 조회되는 랭킹, 통계, 시세 등)
- **Infra & Monitoring**
  - Docker / Docker Compose로 전체 서비스 컨테이너 운영
  - GitHub Actions로 CI/CD 파이프라인 구성
  - Grafana + Prometheus로 백엔드/DB 주요 지표 모니터링

발표 시에는 README의 아키텍처 다이어그램 이미지를 보여주며,  
위 네 줄(프론트/백엔드/DB/인프라)을 짧게 짚어주면 좋습니다.

## 5. 문제와 해결 사례 (Before → After)

이 프로젝트에서는 실제 사용 과정에서 여러 문제가 발견되었고,  
이를 **데이터 분석과 모니터링을 통해 개선**했습니다. (정리: `problem.md`)

### 5-1. 관심 리스트 가격이 0억으로 표시되던 문제

- **문제 (Before)**
  - 일부 아파트의 최근 거래가 없을 때, `current_market_price`가 `None`으로 내려와
  - 프론트엔드에서 `|| 0` 처리로 인해 **0억**으로 표시되는 심각한 UX 문제 발생
- **해결 (After)**
  - 백엔드에서 최근 거래가 없으면,
    - 전체 기간 중 가장 최근 거래,
    - 또는 평균 시세를 활용해 **가격을 보정**
  - 캐시에도 항상 유효한 가격 정보가 들어가도록 로직 수정
- **효과**
  - “0억”으로 보이는 비정상 데이터가 사라지고,
  - 관심 리스트/홈 대시보드의 가격 카드 신뢰도가 크게 향상

---

### 5-2. 실거래 내역에서 면적이 보이지 않던 문제

- **문제 (Before)**
  - 실거래 내역 리스트에서 **면적 컬럼이 누락**되어
  - 사용자가 “어느 면적대의 거래인지” 알 수 없는 상태
- **해결 (After)**
  - 거래 데이터 매핑 시 `area` 필드를 포함하도록 수정
  - UI 그리드를 4열 → 5열(날짜, 구분, 면적, 층, 거래액)로 확장
- **효과**
  - “어떤 면적대에서 이 가격이 형성됐는지”를 직접 확인할 수 있어
  - 상세 페이지의 **의사결정 도움 정도**가 개선

---

### 5-3. 그래프와 실거래 내역이 서로 다른 데이터를 보여주던 문제

- **문제 (Before)**
  - 그래프는 최근 6개월만, 실거래 내역은 더 긴 기간의 데이터를 보여주는 등
  - 같은 화면 안에서 **그래프와 표가 서로 다른 기준**을 사용
- **해결 (After)**
  - 기간/면적/거래유형 필터를 **그래프와 실거래 내역이 공유**하도록 통합
  - 데이터가 부족한 경우, 실제 DB에 있는 기간 전체를 자동으로 탐색하도록 로직 개선
- **효과**
  - “그래프에서 보이는 피크가 실제 거래에도 있는지?” 등  
    그래프와 표를 **교차 검증**할 수 있는 UX로 개선

---

### 5-4. 홈 화면 로딩 속도가 느리던 문제

- **문제 (Before)**
  - 내 자산 각각에 대해 개별 API를 호출해 N개의 요청이 발생
  - 초기 진입 시 로딩 시간이 길어지는 문제
- **해결 (After)**
  - **2단계 로딩 전략** 도입
    1. 기본 정보만으로 먼저 카드 렌더링 (빠른 첫 화면)
    2. 차트/추가 데이터는 백그라운드에서 비동기 로딩
  - 일부 API는 여러 아파트를 한 번에 조회할 수 있도록 **벌크 조회**를 고려/적용
- **효과**
  - 체감 로딩 속도가 크게 개선되고,
  - 홈 진입 시 사용자 이탈 가능성이 줄어듦

## 6. 발표 정리 키 메시지

- **“내 자산”과 “시장 데이터”를 한 서비스 안에서 통합해서 보여준다.**
- **복잡한 부동산 데이터를, 시각화와 비교 기능을 통해 의사결정에 바로 쓸 수 있게 만든다.**
- **실제 문제(0억 표기, 필터 불일치, 느린 로딩 등)를 데이터·아키텍처 개선으로 해결했다.**

위 메시지를 중심으로 1~5번 내용을 연결해서 이야기하면,
